## Context

当前 H5 版 CardLibPage 的“正确的事”卡片已经支持两种高频交互：右划快速打卡、长按打开操作菜单。数据层中也已经保存了完整的 `records` 执行记录（包含 action_id、exec_result、exec_remark、exec_time），但没有任何 UI 入口让用户查看某个动作的详细历史记录。

该需求重点不是新增数据，而是让现有数据可被访问，并且不破坏现有右划/长按交互。

## Goals / Non-Goals

**Goals:**
- 为“正确的事”卡片增加左划查看详情交互
- 提供全屏右侧滑入的详情面板，承载单条动作的完整信息
- 在详情页中展示打卡记录时间线和关联规律
- 明确解决左划、右划、长按、纵向滚动之间的手势冲突
- 卡片常驻轻提示，让用户发现左划功能

**Non-Goals:**
- 不给规律卡片增加左划详情
- 不新增独立路由系统，详情以页面内滑入面板方式实现
- 不扩展到复盘页、首页等其他页面
- 不在第一版加入打卡记录筛选（如只看失败记录）
- 不修改现有打卡数据结构

## Decisions

### 1. 详情使用页面内全屏滑入面板，而非底部弹窗

**选择**：在 CardLibPage 内部维护 `detailAction` 状态，选中后渲染一个 fixed 全屏右侧滑入面板。

**替代方案**：底部弹出大弹窗。

**理由**：打卡记录列表天然是纵向长内容，全屏面板更适合承载“概览 + 时间线 + 关联规律”三段式结构。底部弹窗更适合轻量确认类内容，不适合长期停留和滚动浏览。

### 2. 左划与右划共用同一手势系统

**选择**：ActionCard 内部统一处理 horizontal swipe。以触摸起点和终点的 X/Y 位移计算：
- `abs(dx) > abs(dy)` 且 `dx < -threshold` → 左划详情
- `abs(dx) > abs(dy)` 且 `dx > threshold` → 右划打卡
- 水平位移达阈值后取消长按计时

**理由**：如果左划和右划分散在不同逻辑里，很容易出现手势判断重叠。统一判定能更清晰地处理优先级。

### 3. 长按仅在“低位移”条件下触发

**选择**：只有在触摸持续 500ms 且横向/纵向位移都小于小阈值时，才触发长按菜单。

**理由**：避免用户试图左划时被误判成长按。长按是低频操作，优先级应低于明确的左右滑动作。

### 4. 详情数据按“动作 + records + laws”聚合计算

**选择**：详情面板不新增单独的复杂数据模型，直接用：
- `action`：当前动作本身
- `records`：从 mockApi.records 中筛出 `action_id === action.id`，按 `exec_time` 倒序排列
- `relatedLaws`：从 laws 中筛出 `related_action_id === action.id`

**理由**：数据已存在，聚合即可，避免额外冗余状态。

### 5. 卡片提示采用“常驻箭头 + 辅助浅灰文案”双提示

**选择**：卡片右侧常驻小箭头 `›`，卡片底部增加浅灰提示文案“左划查看详情”。

**替代方案**：仅箭头、不加文案；或首次进入时动画引导。

**理由**：箭头单独存在时语义不够明确，文案能立即建立用户心智。相比一次性动画，常驻提示更稳定。

## Risks / Trade-offs

- **手势冲突复杂** → 需要精细处理 horizontal vs vertical 位移判断。Mitigation: 采用“水平位移显著大于垂直位移”再进入左右滑逻辑。
- **详情面板层级与现有 modal/action sheet 叠加** → fixed 层可能覆盖不当。Mitigation: 详情页使用独立 z-index，打开详情时不允许同时打开菜单/打卡弹窗。
- **记录很多时滚动性能** → 当前 mock 数据量小，可接受。后续若记录量很大，再考虑虚拟列表。
