## Context

H5 版方法论进化器采用 React + Vite 单页应用架构，底部 5 Tab 切换，数据层使用 mock.js 模拟 API。当前 4 个页面（首页、卡片库、SOP、复盘）已完成，均遵循相同模式：从 `mock.js` 导入 `api` 对象，组件内部管理状态，样式写在全局 `App.css` 中。

个人中心是最后一个待实现页面，无跨模块依赖，不涉及路由改造。

## Goals / Non-Goals

**Goals:**
- 实现完整的个人中心页面，与其他 4 个页面风格一致
- 分类管理支持内联 CRUD（新增、改名、排序），无需跳转二级页面
- 设置项状态持久化到 mock 数据层（页面刷新后保持）
- 数据管理提供完整的模拟交互反馈

**Non-Goals:**
- 不实现真实的深色模式 CSS 切换（仅 UI 占位）
- 不实现真实的云端备份/还原（仅模拟流程）
- 不实现隐私密码设置（需求文档标注为后续迭代）
- 不新增路由系统或页面跳转能力

## Decisions

### 1. 分类管理采用页面内展开折叠

**选择**：点击"分类管理"行后在当前页面内展开分类列表，支持内联编辑。

**替代方案**：全屏 Modal 弹窗。

**理由**：H5 版无路由系统，全屏 Modal 会打断页面滚动位置且实现复杂度更高。内联展开更轻量，符合"设置页"的操作习惯。

### 2. 深色模式使用三档 Radio 选择器

**选择**：三个选项（浅色/深色/跟随系统）以 radio group 形式展示，选中后存入 mock 状态但不触发实际主题切换。

**理由**：需求文档明确要求三档模式，用 switch 开关无法表达"跟随系统"。UI 先到位，后续接入 CSS 变量时只需读取这个状态值。

### 3. 数据概览放在用户信息卡片下方

**选择**：用户信息卡片 → 数据概览条 → 设置列表，自上而下。

**理由**：让页面顶部有"成就感"内容，避免纯设置列表的冷淡感。数据从现有 mock 中聚合计算，无额外 API。

### 4. 排序使用上移/下移按钮

**选择**：分类列表每行提供 ↑↓ 按钮调整顺序。

**替代方案**：拖拽排序。

**理由**：与 SOP 页面步骤排序保持一致的交互模式，且移动端拖拽实现复杂、体验不稳定。

## Risks / Trade-offs

- **Mock 数据无持久化** → 页面刷新后分类编辑和设置变更会丢失。这是 H5 演示版的已知限制，可接受。后续接入真实后端时自然解决。
- **样式膨胀** → 所有页面样式集中在 App.css，文件会继续增长。当前阶段可接受，后续可拆分为 CSS Modules。
- **无表单校验库** → 分类新增/改名的校验（非空、长度限制）手写实现。当前页面简单，不值得引入额外依赖。
