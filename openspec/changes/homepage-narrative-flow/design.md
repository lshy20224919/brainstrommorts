## Context

当前首页结构为：数据看板（三列统计）→ 快捷操作区（四按钮）→ 待办提醒 → 榜单。用户看到的是并列的数据，感受不到产品"记录→验证→提炼→固化→迭代"的递进逻辑。操作完成后只有 Toast，没有引导用户进入下一个阶段的衔接。

## Goals / Non-Goals

**Goals:**
- 首页第一屏让用户 3 秒内理解"我在方法论进化的哪个阶段"
- 智能建议基于真实数据动态生成，不是写死文案
- 操作后的引导自然不打扰，用户可以选择跟随或忽略
- 保持极简风格，不引入复杂动画或过多视觉元素

**Non-Goals:**
- 不做真正的 AI 推荐算法，用规则引擎即可
- 不改变 Tab 结构或路由系统
- 不改动卡片库、SOP、复盘、个人中心的核心逻辑
- 不做强制引导或新手教程弹窗

## Decisions

### 1. 进化旅程用"5 步进度条"而非环形图

**选择**：横向 5 步进度条，每步一个圆点 + 标签 + 数据。已达成的步骤实心深色，未达成的空心浅色。

**替代方案**：环形进度图、雷达图。

**理由**：5 步进度条最直观地传达"线性递进"关系，用户一眼就能看到"我走到第几步了"。环形图更适合单一指标的完成度，雷达图适合多维对比，都不如线性进度条直观表达"阶段递进"。

### 2. 阶段达成判定使用简单阈值规则

**选择**：
- 阶段 1（记录）：有 ≥1 条正确的事 → 达成
- 阶段 2（验证）：总打卡次数 ≥5 → 达成
- 阶段 3（提炼）：有 ≥1 条规律 → 达成
- 阶段 4（固化）：有 ≥1 个 SOP 模板 → 达成
- 阶段 5（迭代）：有 ≥1 条复盘记录 → 达成

**理由**：简单明确，用户容易理解"我做了什么就能点亮下一步"。不需要复杂的加权计算。

### 3. 智能建议使用优先级规则引擎

**选择**：按优先级从高到低检查条件，返回第一条匹配的建议：
1. 无任何正确的事 → "记录你做对的第一件事"
2. 有正确的事但从未打卡 → "去执行一次，验证它是否有效"
3. 打卡 ≥5 次但无规律 → "从成功经验中提炼规律"
4. 有规律但无 SOP → "把规律组合成可执行的流程"
5. 有 SOP 但无复盘 → "做一次复盘，看看方法论是否在进化"
6. 全部达成 → 推荐高频动作的具体优化建议（如"XX 成功率高，建议固化"）

**理由**：规则引擎简单可控，不需要 ML 模型，且每条建议都有明确的跳转目标。

### 4. 操作后引导使用"底部浮层"而非弹窗

**选择**：打卡成功后在页面底部弹出一个轻量浮层（类似 Snackbar），展示 1-2 个操作按钮，3 秒后自动消失或用户点击后消失。

**替代方案**：Modal 弹窗。

**理由**：Modal 太重，打断感强。底部浮层轻量、不遮挡内容、用户可以忽略，符合"自然不打扰"的设计原则。

### 5. SOP 执行汇总使用全屏结果页

**选择**：SOP 全部步骤完成并提交后，展示一个全屏结果页，列出本次执行的所有动作及其成功/失败状态，底部有"完成"按钮返回。

**理由**：SOP 执行是一个完整的"仪式"，结束时给一个汇总页能产生成就感闭环，比 Toast 有意义得多。

## Risks / Trade-offs

- **首页信息密度增加** → 进化旅程 + 建议 + 快捷操作 + 待办，可能显得拥挤。Mitigation: 建议卡片只显示一条，待办区无数据时隐藏。
- **建议逻辑可能不准** → 规则引擎是静态的，可能推荐不够智能。Mitigation: 第一版先覆盖核心路径，后续可迭代规则。
- **操作后浮层可能被忽略** → 用户可能不注意底部浮层。Mitigation: 浮层有轻微弹入动画，且按钮颜色醒目。
